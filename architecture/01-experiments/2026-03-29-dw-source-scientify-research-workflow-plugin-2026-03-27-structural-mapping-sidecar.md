## structural mapping

- `problem_class`: mixed-value source partition between transferable Runtime capability and non-transferable plugin/host shell
- `source_structure`: callable tools and skills sit inside a larger plugin shell with orchestration, hooks, scheduled delivery, and host-specific runtime assumptions
- `target_structure`: Directive-owned bounded Runtime capability should keep the callable literature-access core while excluding the plugin shell and host-specific execution baggage
- `relational_correspondence`: the strongest match is not "multi-step research workflow" but the relation between a reusable capability core and the surrounding plugin/runtime shell that should remain outside the retained Runtime slice
- `transferable_relation`: isolate the callable capability core from the source-specific plugin and host shell
- `control_pattern`: a bounded capability can be promoted while the surrounding operational shell remains explicitly unopened
- `transformation_pattern`: reimplement callable literature-access tools as Directive-owned Runtime capability while leaving plugin registration, scheduling, and host transport behind
- `evidence_pattern`: proof and promotion-readiness artifacts are used to keep the retained capability inspectable before any execution or host integration
- `transfer_conditions`:
  - retain only the bounded callable tool bundle
  - preserve explicit proof and promotion gates before any host-facing move
  - keep Runtime ownership on lifecycle truth and capability legality
- `non_transfer_conditions`:
  - do not copy OpenClaw plugin registration
  - do not copy cron or hook infrastructure
  - do not copy gateway/runtime transport assumptions
  - do not treat the full Scientify workflow shell as automatically reusable Runtime capability
- `source_specific_baggage`:
  - plugin API and registration surfaces
  - scheduled delivery and hook injection
  - OpenClaw-specific runtime and gateway assumptions
  - broader research-pipeline orchestration beyond the retained 4-tool slice
- `false_analogy_risk`: high if the source is read as "Directive Workspace should adopt a research-workflow plugin system"; low if the source is read as a capability-core versus shell partition case
- `blueprint_confidence`: high
