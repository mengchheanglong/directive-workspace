# AGENTS.md - Directive Workspace Operating Guide

This file applies specifically to `directive-workspace/`.

## Start here

Before substantive work in this folder:

1. Read `CLAUDE.md` in this folder first.
2. Treat it as the governing doctrine for Directive Workspace.
3. Use the relevant skills from `.claude/skills` when they match the task:
   - `directive-workspace-doctrine`
   - `directive-workspace-audit`

## Core operating rule

Directive Workspace is the product.
Engine is the shared adaptation core inside it.
Discovery, Forge, and Architecture are the three main operating lanes of the Engine.

Do not treat those lanes as loose peer products.
Do not collapse Engine into Architecture.

## Priority rules

Always prioritize:
- Engine as the shared adaptation core
- Discovery as goal-aware intake, filtering, and routing
- Forge as reusable runtime usefulness conversion
- Forge as behavior-preserving transformation
- Architecture as Engine self-improvement / operating code
- mission-conditioned usefulness
- proof, decision, integration, and reporting discipline
- reversible, measurable changes

## Working rule

Prefer work that makes Directive Workspace more real as an operating system for source adaptation:
- less scattered shared logic
- clearer Engine ownership
- clearer lane boundaries
- better routing
- better proof/evaluation
- better source-to-usefulness flow

Avoid:
- reducing the product to repo intake only
- reducing Forge to tool adoption only
- reducing Architecture to passive notes
- dropping Decide or Report from the workflow
- broad speculative framework expansion without workflow pressure

## Execution rule

For major changes, state:
- affected layer
- owning lane
- mission usefulness
- proof path
- rollback path

Run the relevant checks after meaningful change groups.
Keep changes reversible and architecture-first.
